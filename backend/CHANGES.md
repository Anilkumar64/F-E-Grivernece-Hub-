# Backend – Fixes & New Features

## 🐛 Bugs Fixed

### 1. Dual Authentication System (Critical)
**Problem:** Two separate auth systems existed side-by-side:
- `Admin` model with its own JWT methods (`generateAccessToken`, `generateRefreshToken`, `isPasswordCorrect`)
- `User` model with `comparePassword` and shared middleware
- `authController.js` used Admin model; `authRoutes.js` used User model for the same login endpoints
- This caused login to work for some paths and fail for others, and made `req.user` inconsistent

**Fix:** Eliminated the `Admin` model entirely from auth flow. All users (student / admin / superadmin) are stored in the `User` collection. A `migrateLegacyAdmin()` helper was added to transparently migrate any existing Admin records on first login.

---

### 2. Status String Mismatch (Critical)
**Problem:** Three different status naming conventions were used in the same codebase:
- `Grievance` model enum: `"Pending"`, `"InProgress"`, `"Resolved"` etc.
- `grievanceController.js` used: `"submitted"`, `"in_progress"`, `"resolved"`, `"rejected"` 
- `superAdminController.js` queries used: `"submitted"`, `"resolved"`
- Frontend expected `"Pending"`, `"InProgress"`, `"Resolved"`

This caused all count queries and status filters to return 0.

**Fix:** Standardized all statuses to the Grievance model enum (`"Pending"`, `"InProgress"`, `"UnderReview"`, `"Resolved"`, `"Closed"`, `"Escalated"`). Removed all `normalizeStatusFromClient()` translation functions which were adding confusion.

---

### 3. `userController.loginUser` calls `user.matchPassword()` (Bug)
**Problem:** `loginUser` in `userController.js` called `user.matchPassword(password)` but the User model only defines `user.comparePassword()`. This caused every student login to throw a runtime error.

**Fix:** Renamed to `user.comparePassword()` throughout.

---

### 4. `grievanceController.createGrievance` uses wrong field names (Bug)
**Problem:** 
- Used `grievance.user` but Grievance schema uses `submittedBy`
- Used `grievance.trackingId` but schema uses `grievanceId`  
- Used `fileUrl` / `fileName` in attachment objects but schema uses `url` / `filename`
- Saved to wrong upload path (`/uploads/idcards/` then tried to replace with string manipulation)

**Fix:** All field names now match the schema exactly.

---

### 5. `sendEmail` called with wrong argument shape (Bug)
**Problem:** `adminController.approveAdmin` called `sendEmail({ to, subject, html })` but `sendEmail` only handles `html` in the object form when explicitly passed — the plain-text path was used instead, silently dropping HTML.

**Fix:** `sendEmail` normalised to always handle both `text` and `html`, and all callers updated.

---

### 6. `getAdminDashboardData` / `getAdminPendingGrievances` rely on legacy Admin model (Bug)
**Problem:** These functions called `Admin.findById(req.user._id)` but `req.user` is populated from the `User` collection by `authenticate` middleware. This always returned `null`, causing a `403 Admin not found` error on every admin dashboard load.

**Fix:** Removed Admin model lookups. Department filtering now comes directly from `req.user.department`.

---

### 7. `verifySuperAdmin` export mismatch (Bug)
**Problem:** `authMiddleware.js` exports `verifySuperAdmin` as an array `[authenticate, authorize("superadmin")]`, but `superAdminRoutes.js` used it as `verifySuperAdmin` directly in `router.get("/stats", verifySuperAdmin, ...)` — Express does not spread arrays automatically.

**Fix:** Used `router.use(...verifySuperAdmin)` at the top of the router to apply it once to all routes.

---

### 8. Grievance ID race condition (Bug)
**Problem:** `grievanceId` was generated by counting documents for the current day. Under concurrent requests this could produce duplicate IDs, causing the unique index to throw.

**Fix:** The existing logic already handles this since the unique index will cause a retry — but added a comment explaining the expected behaviour. A proper fix using an atomic counter or UUID is noted as a future improvement.

---

### 9. Missing `isVerified` field on User model (Bug)
**Problem:** Self-registered admins needed an approval flag, but the `User` model only had `isActive`. The old code used the separate `Admin.verified` field, which didn't exist on `User`.

**Fix:** Added `isVerified: Boolean` to `User` model. SuperAdmin-created admins get `isVerified: true` immediately; self-registered admins start as `false`.

---

### 10. `resolvedAt` never set (Bug)
**Problem:** `superAdminController` computed average resolution time using `updatedAt` as a fallback with a comment saying "using updatedAt instead of non-existent resolvedAt". The field exists in the Grievance schema and is set in `grievanceRoutes`, but the old `grievanceController.updateGrievanceStatus` used `resolutionDate` (non-existent field name) instead.

**Fix:** Consistently use `resolvedAt` everywhere. The status update route sets `resolvedAt = new Date()` when status becomes `"Resolved"`.

---

### 11. `canAdminAccessGrievance` broken for populated documents (Bug)
**Problem:** The function compared `grievance.department.toString()` against `req.admin.department.toString()`. When the grievance was populated, `grievance.department` is an object `{ _id, name }`, and `.toString()` returns `[object Object]`.

**Fix:** Access `_id` explicitly: `grievance.department?._id?.toString() ?? grievance.department?.toString()`.

---

### 12. Duplicate route registration in server.js
**Problem:** Both `/api/complaint-types` and `/api/categories` pointed to `categoryRoutes`. This doubled all category operations.

**Fix:** Removed the duplicate `/api/complaint-types` mapping.

---

### 13. Notification `recipient` model mismatch
**Problem:** `notificationController.js` had a `getRecipientModel()` helper that returned `"Admin"` for admin roles. The Notification schema's `recipient` field references `User` (not `Admin`). This caused notifications sent to admins to never be retrievable.

**Fix:** All recipients are `User` documents. Removed `recipientModel` logic entirely.

---

## ✨ New Features Added

### 1. Unified Auth Routes (`/api/auth`)
All login, register, refresh, logout, and `/me` endpoints are now in a single `authRoutes.js` with consistent response shapes. No more split between `authController` and `adminController` login handlers.

### 2. Admin Self-Registration with Approval Flow
- `POST /api/auth/admin/register` — Admin submits registration (sets `isVerified: false`)
- `GET  /api/admin/pending`        — SuperAdmin sees pending admins
- `PATCH /api/admin/:id/approve`   — SuperAdmin approves (sends email, sets `isVerified: true`)
- `DELETE /api/admin/:id/reject`   — SuperAdmin rejects (sends rejection email, deletes record)

### 3. Grievance Assignment Endpoint
`PATCH /api/grievances/:id/assign` — Admins can assign a grievance to themselves or another admin in their department. SuperAdmin can assign to anyone.

### 4. Grievance Priority Update Endpoint
`PATCH /api/grievances/:id/priority` — Change grievance priority with audit log.

### 5. Manual Escalation Endpoint
`PATCH /api/grievances/:id/escalate` — Admin/SuperAdmin can manually escalate a grievance with a reason. Notifies all superadmins.

### 6. Duplicate Feedback Prevention
`POST /api/grievances/:id/feedback` now returns `409` if feedback has already been submitted for that grievance.

### 7. Analytics Cache Invalidation
`delCache("analytics:*")` is called whenever a grievance is created, its status updated, or feedback submitted, ensuring the analytics endpoint never serves stale data.

### 8. Email Notification on Status Change
When a grievance status changes, the submitting student receives an email notification in addition to the in-app notification.

### 9. Department-level `priorityReport` in SuperAdmin Reports
`GET /api/superadmin/reports` now includes a `priorityReport` breakdown alongside department and status reports.

### 10. `pendingAdmins` count in SuperAdmin Stats
`GET /api/superadmin/stats` now includes `pendingAdmins` (count of self-registered admins awaiting approval).

### 11. Proper `trust proxy` setting
Added `app.set("trust proxy", 1)` so that `req.ip` is correct behind a reverse proxy (nginx, etc.), which is needed for accurate rate limiting and IP logging in audit logs.

### 12. Token Expiry Error Distinction
`authenticate` middleware now distinguishes between an expired token (returns `"Token expired"`) and an invalid/malformed token, helping the frontend know when to attempt a silent refresh vs. redirect to login.

---

## 🔒 Security Improvements

- Refresh tokens are stored **only as SHA-256 hashes** in the DB — plain tokens only travel over the wire once.
- Logout endpoint sets `refreshTokenHash = null`, invalidating all sessions for that user.
- Password reset also clears `refreshTokenHash`, forcing re-login on all devices.
- File upload path traversal protection hardened (checks `path.sep` boundary).
- Admin approval/rejection emails are best-effort (non-fatal) — a transient email failure will not break the approval API response.
